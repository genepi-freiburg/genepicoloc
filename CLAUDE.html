CLAUDE.md - genepicoloc Package Development

Project Overview

genepicoloc is an R package for genetic colocalization analysis across multiple QTL and PheWAS datasets. It wraps the coloc package with scalable infrastructure for large-scale analyses.

Repository: https://github.com/genepi-freiburg/genepicoloc

Production Use Case: CKDGen Consortium

The package powers a large-scale colocalization analysis for the CKDGen Consortium (kidney function GWAS). Key achievements:

Scale

  • 7 CKDGen traits: eGFRcr, eGFRcys, UACR, Calcium, Phosphate, Albumin, Urate
  • 9 external data sources: GTEx eQTL, eQTLGen, UKB_PPP (proteomics), Icelanders pGWAS, FinnGen, MVP, UKB_TOPMed, GCKD metabolomics, CKDGen internal
  • ~200,000+ colocalization tests per study
  • Results stored efficiently in tar archives with job-based parallelization

Shiny Visualization App

A production-ready Shiny app was developed: ckdgen-coloc-app/

Features: - Network visualization (region-centered and trait-centered views) - Interactive filtering by PP.H4 threshold and study - Regional Association Plots (LocusZoom-style) with interactive plotly - Gene Info Panel with annotations from NCBI, Reactome, HPO, HGNC - AI-powered insights (Claude API integration) - Docker deployment ready

Key files to reference: - /home/oleg/Work/bioinfo/ckdgen/git/followup/genepicoloc/ckdgen-coloc-app/ - Full Shiny app - /home/oleg/Work/bioinfo/ckdgen/git/followup/genepicoloc/ckdgen_genepicoloc_functions.R - CKDGen-specific helper functions

Regional Association Plots (RAPs)

Successfully implemented regional plot data extraction:

Workflow: 1. Run genepicoloc with save_sumstats=TRUE to save sumstats_2 RDS files in tar archives 2. Extract regional data using optimized job-grouped extraction (5-10x faster) 3. Serve in Shiny app with interactive LocusZoom-style plots

Key optimizations discovered: - Group regions by job, extract tar once per job (not per region) - Deduplicate trait names before processing - Use filename for Icelanders_pGWAS (short name annotation has errors)

GCKD VAE Analysis

Second use case: colocalization for GCKD multi-biomarker GWAS (VAE-derived modules).

Input

  • ME19 and ME30 modules, SNPTEST format (.gwas), hg19, N=4,850
  • Severe p-value underflow: lead SNPs have p ~ 1e-1762 (z ~ 90)
  • pvalue_precise_normal column preserves full precision as text

Preprocessing Pipeline

  1. Load .gwas file with fread()
  2. Handle underflow: handle_underflow(pvalue_precise_normal, return_nlog10P = TRUE)
  3. LiftOver hg19 -> hg38: genepi_liftover() + setDT() on result
  4. Column mapping: SNP->rsID, coded_all->A1, noncoded_all->A2, AF_coded_all->AF, n_total->N
  5. Build Name: chr:pos:ref:alt format

Results (MVP EUR only)

  • ME19: 1 region (chr2/NAT8), 7 high-confidence colocalizations (creatinine, eGFR, Mg, CKD, BUN, AST)
  • ME30: 2 regions (chr10/HPSE2 lead signal nlog10P=3274.8, chr2/NAT8 secondary), 12 high-confidence colocalizations
  • All colocalizations at chr2/NAT8 locus - kidney function signal
  • chr10/HPSE2 region: no MVP colocalizations (may colocalize with eQTL/pQTL)

Files

  • Analysis script: genepicoloc_internal/analyses/GCKD_VAE/run_GCKD_VAE_coloc.R
  • Output: /g/epi/data/studies/00_GCKD/01_analyses/gwas/multi_biomarker_GWAS/common_chip/21_VAE/02_output/genepicoloc/

HPC Architecture Decisions

Parallelization Strategy: SLURM Array Jobs

  • Decision: Use SLURM array jobs for multi-node parallelization (not Nextflow/Snakemake)
  • Rationale: Workload is embarrassingly parallel (no inter-job dependencies), SLURM is already available, zero new dependencies
  • Approach (Option B - cleaner separation):
    1. genepicoloc_prepare() - new function to save inputs and return job count
    2. sbatch --array=1-N - SLURM submits jobs in parallel
    3. genepicoloc_job_standalone(dir_out, job_idx) - each array task runs one job
    4. consolidate_coloc_results() - unchanged, runs after all jobs complete
  • Status: Not yet implemented, planned as next major improvement

Future I/O Optimizations (lower priority)

  • Batch tabix queries (1 call for all regions per file instead of per-region) - ~2-5x speedup
  • Pre-filter empty datasets before main run - variable speedup

Preprocessing Functions

genepi_liftover()

Converts genomic coordinates from hg19 to hg38 using UCSC liftOver.

Location: R/name_by_position.R

Usage:

sumstats_hg38 <- genepi_liftover(
  sumstats = sumstats,
  CHR_name = "Chr",
  POS_name = "Pos_b37",
  A1_name = "Allele1",
  A2_name = "Allele2",
  liftOver_bin = "/path/to/liftOver",
  liftOver_chain = "/path/to/hg19ToHg38.over.chain.gz"
)

Output columns: CHR_hg38, POS_hg38, A1_hg38, A2_hg38

Features: - Works on a copy (doesn’t modify input) - Proper temp file handling with on.exit() cleanup - Auto-detects chr prefix format - Handles strand flipping during liftOver

Required tools: - UCSC liftOver binary: https://hgdownload.soe.ucsc.edu/admin/exe/linux.x86_64/liftOver - Chain file: https://hgdownload.soe.ucsc.edu/goldenPath/hg19/liftOver/hg19ToHg38.over.chain.gz

name_by_position()

Adds rsID and standardized variant names by querying Ensembl VCF files.

Location: R/name_by_position.R

Usage:

# Option 1: Single VCF file (e.g., chr21 only for testing)
sumstats_named <- name_by_position(
  sumstats = sumstats_hg38,
  CHR_name = "CHR_hg38",
  POS_name = "POS_hg38",
  A1_name = "A1_hg38",
  A2_name = "A2_hg38",
  dbSNP_file = "/path/to/homo_sapiens-chr21.vcf.gz"
)

# Option 2: Per-chromosome VCF directory (recommended for production)
sumstats_named <- name_by_position(
  sumstats = sumstats_hg38,
  CHR_name = "CHR_hg38",
  POS_name = "POS_hg38",
  A1_name = "A1_hg38",
  A2_name = "A2_hg38",
  dbSNP_dir = "/path/to/ensembl_vcf/"  # contains homo_sapiens-chr{1..22,X,Y,MT}.vcf.gz
)

Parameters: - dbSNP_file: Path to single tabix-indexed VCF file - dbSNP_dir: Path to directory with per-chromosome files (homo_sapiens-chr{CHR}.vcf.gz)

Output columns: rs (rsID), Name_hg38 (chr:pos:ref:alt format)

Features: - Works directly with Ensembl VCF files (no preprocessing needed) - Supports both single-file and per-chromosome directory modes - Auto-detects chromosome naming convention (chr21 vs 21) per file - Handles multi-allelic variants (splits comma-separated ALT alleles) - Matches alleles accounting for strand flipping - Skips chromosomes without corresponding VCF file (warns user) - No external dependencies (uses system2 + tabix)

Ensembl VCF source:

# For testing (chr21 only, ~165MB):
wget https://ftp.ensembl.org/pub/release-115/variation/vcf/homo_sapiens/homo_sapiens-chr21.vcf.gz
wget https://ftp.ensembl.org/pub/release-115/variation/vcf/homo_sapiens/homo_sapiens-chr21.vcf.gz.csi

# For production (all chromosomes, ~15GB total):
for chr in {1..22} X Y MT; do
  wget https://ftp.ensembl.org/pub/release-115/variation/vcf/homo_sapiens/homo_sapiens-chr${chr}.vcf.gz
  wget https://ftp.ensembl.org/pub/release-115/variation/vcf/homo_sapiens/homo_sapiens-chr${chr}.vcf.gz.csi
done

Performance (chr21): - Input: 112,722 variants - Matched: 112,722 variants (100%) - Processing time: ~2 minutes

# 4. Calculate -log10(P) with underflow handling
sumstats_named[, nlog10P := -log10(P)]
sumstats_named[!is.finite(nlog10P), nlog10P := max(nlog10P[is.finite(nlog10P)]) + 1]

Technical Notes

Tar-based Storage

Results are stored in tar archives per job: - jobs/job_XXX/coloc_results.tar - Colocalization results - jobs/job_XXX/sumstats.tar - Sumstats for regional plots (if save_sumstats=TRUE) - jobs/job_XXX/sumstats_1.RDS - Source study sumstats

Annotation Pipeline

After colocalization, coloc_out_annotate() in genepicoloc_internal: 1. Reads consolidated results from dir_out/unfilt/ 2. Dispatches to study-specific annotator: - annotate_eQTL() - GTEx, eQTLGen, Kidney_eQTL (uses gencode_v26 for gene mapping) - annotate_pQTL() - UKB_PPP, Icelanders, GCKD_pGWAS (uses olink protein map) - annotation_std() - GWAS/PheWAS studies (uses annotation DB for phenotype names) 3. Writes annot/study_annot_unfilt.RDS 4. Filters in 3 tiers: coloc_done -> PP.H4 > 0.8 -> sumstats_2 genome-wide sig 5. Final: annot/annot_filt.RDS (also .xlsx)

Known Issues & Solutions

genepi_liftover() returns data.frame - Returns base R data.frame, not data.table - Must call setDT() on result before using data.table syntax (:=, bare column names in [)

MVP annotation DB has old /data/ paths - annotation_db.RDS was created with old /data/ mount paths - Fix in arguments_custom.R: sub("^/data/", "/g/epi/data/", ...) on file paths from DB - Proper fix: regenerate annotation DB with create_annotation_db()

MVP ancestry-split annotation - MVP_R4_AFR/AMR/EAS/EUR/META studies share the MVP_R4 annotation DB entry - annotation_std() maps grepl("^MVP_R4_", study) to “MVP_R4” for DB lookup

P-value underflow handling - Some GWAS have p-values far beyond R’s numeric range (e.g., 1e-1762 for GCKD VAE) - handle_underflow() parses scientific notation text strings directly - Fallback from z-scores: nlog10P = -log10(2) - pnorm(abs(z), lower.tail=FALSE, log.p=TRUE) / log(10) - Always check for text-format p-value columns (e.g., pvalue_precise_normal in SNPTEST output)

MVP variant naming bug (historical) - Old MVP data had mismatched variant names - Solution: Always verify nsnps is reasonable (~6000, not ~20)

Non-positive definite LD matrices (coloc.susie) - Reference panel mismatch causes negative eigenvalues - Solution: Eigenvalue shift method in shift_eigenvalues_minimal()

Icelanders_pGWAS protein naming - Short name annotation has errors (wrong proteins labeled) - Solution: Use filename (contains SeqId + protein name)

UKB_PPP protein complex duplicates - Multi-protein assays (e.g., FUT3_FUT5) create duplicate entries - Expected behavior for comparison functions

Data Concepts

sumstats_1 vs sumstats_2

  • sumstats_1: Your GWAS of interest (e.g., eGFR, UACR) - the “query”
  • sumstats_2: External datasets to colocalize against (eQTL, pQTL, PheWAS) - the “targets”

Required Columns

  • Name: CHR:POS:REF:ALT format (e.g., chr16:17906244:T:C)
  • CHR, POS: Genomic coordinates (GRCh38)
  • A1, A2: Effect and other allele
  • BETA, SE: Effect size and standard error
  • nlog10P: -log10(p-value) to avoid underflow
  • AF: Allele frequency
  • N: Sample size

coloc.susie Notes (Advanced Fine-mapping)

For studies needing SuSiE-based fine-mapping:

EUR vs ALL Ancestry

  • EUR ancestry recommended for predominantly European GWAS
  • ALL ancestry suffers from LD heterogeneity → weak/scattered credible sets
  • EUR shows strong colocalization (PP.H4 ~0.97) where ALL shows none (~0.001)

Direction Filtering

  • Filter variants by “Direction” column (from meta-analysis)
  • Keep variants present in most studies (≤2 “?” allowed)
  • Dramatically improves LD matrix eigenvalues

Reference Allele Alignment

  • Critical: use fixref=TRUE in LD matrix creation
  • Aligns effect allele from sumstats to LD reference panel
  • Without this, SuSiE fails to converge
